# Frontend Guide: Nango OAuth Multi-Account Integration (Lucid)

This document explains how the frontend should integrate with the **Express offchain API** to support:
- multiple connected accounts per provider (e.g. 2 Twitter accounts)
- dynamic dropdown options (lists/bases/channels/etc.) via Nango Proxy
- workflow execution calls scoped to the **selected account** via `connectionId`

## Key concept: `connectionId`

In this system:
- **`connectionId` is the Nango connection id** used when connecting an account.
- It is generated by our backend during `/initiate`.
- It must be persisted in node configuration and passed back to backend endpoints.

Example format:
```
${privyUserId}-${provider}-${uuid}
```

> Do not confuse this with the DB row `id` in `user_oauth_connections`. For frontend purposes, treat `connectionId` as the “credential id”.

---

## Endpoints you will call

Base: `https://api.lucid.foundation/api/oauth` (or local dev base)

### 1) List available providers
`GET /providers`

Returns providers configured in Nango.

---

## Multi-account OAuth connect flow (required)

### Step A — initiate connect (backend generates `connectionId`)
`POST /:provider/initiate`

Auth: **Privy JWT** (`Authorization: Bearer <token>`)

Response includes:
```json
{
  "authUrl": "https://<NANGO_PUBLIC_URL>/oauth/connect/<integrationId>?connection_id=...",
  "connectionId": "did:privy:...-twitter-<uuid>",
  "provider": "Twitter / X",
  "scopes": ["..."],
  "expiresIn": 300
}
```

**Frontend action:**
- Open `authUrl` in popup/new window.
- Store the returned `connectionId` in memory for the next step.

> We support multiple accounts because every connect attempt gets a unique `connectionId`.

### Step B — after Nango flow succeeds, sync metadata into our DB
`POST /:provider/sync`

Body:
```json
{ "connectionId": "<connectionId from initiate>" }
```

Response:
```json
{
  "success": true,
  "provider": "twitter",
  "privyUserId": "did:privy:...",
  "connectionId": "...",
  "profile": {
    "username": "...",
    "displayName": "...",
    "avatarUrl": "..."
  }
}
```

**Frontend action:**
- On success, refresh the connections list.

### Step C — list all connections for the user
`GET /connections`

Auth: Privy JWT

Returns an array with provider + profile fields.

**Important fields:**
- `provider`
- `providerUsername`, `providerDisplayName`, `providerAvatarUrl`

> If you need the `connectionId` itself for selection, we currently expose it indirectly as `nango_connection_id` in DB; we can optionally surface it explicitly in this endpoint if the UI needs it. (Right now, UI can keep the `connectionId` it created/used; for older connections, we may want to add it to the response.)

## Dynamic dropdown options (Priority 1)

### Endpoint
`GET /:provider/resources/:resource?connectionId=...&<optionalParentParams>`

Auth: Privy JWT

Example (Twitter lists):
`GET /twitter/resources/lists?connectionId=<connectionId>`

Example (Airtable tables requires baseId):
`GET /airtable/resources/tables?connectionId=<connectionId>&baseId=app123`

Response:
```json
{ "options": [{ "name": "Option label", "value": "OptionId" }] }
```

**Frontend guidance:**
- Any “dynamic options” hook should include the selected credential’s `connectionId`.
- For cascading dropdowns, pass parent selections as query params (e.g. `baseId`).

---

## Execution-time calls scoped to selected account

### Token (for n8n or client-side proxy consumer)
`GET /:privyUserId/:provider/token?connectionId=<connectionId>`

Auth: HMAC signature (used by n8n). This is typically not called by the UI.

### Backend proxy (if UI/server triggers a one-off action)
`POST /:privyUserId/:provider/proxy`

Body:
```json
{
  "endpoint": "/2/tweets",
  "method": "POST",
  "data": { "text": "hello" },
  "connectionId": "<connectionId>"
}
```

If `connectionId` is omitted, backend uses “most recently used” connection for backward compatibility.

---

## Disconnecting accounts

### Revoke a specific connection
`DELETE /:provider`

Body:
```json
{ "connectionId": "<connectionId>" }
```

If `connectionId` is omitted, backend revokes “most recently used”.

**Frontend guidance:**
- Always send `connectionId` when user clicks “Disconnect” on a specific account.

---

## What frontend must persist in node configuration

For every OAuth-backed node/action/trigger, store:
- `provider` (e.g. `twitter`)
- `connectionId` (the chosen account)
- any dynamic selections (e.g. `listId`, `baseId`, `channelId`)

This ensures:
- dynamic options fetch uses correct account
- execution uses correct account

## Notes / known follow-ups

1) **Expose `connectionId` in `/connections` response**
   - Today `listConnections()` returns DB row fields but not explicitly `nango_connection_id`.
   - If the UI needs to show/select among existing connections without having cached the original value, we should update backend response to include `connectionId: conn.nango_connection_id`.

2) **Profile fetching**
   - Twitter profile is currently fetched after sync (best effort) so avatars/names appear.
   - Additional providers can be added similarly.

3) **Callback endpoint**
   - The offchain `/callback` exists but the primary supported approach is **Nango-hosted flow + `/sync`**.
